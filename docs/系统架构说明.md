# 系统架构说明

## 核心结论

本项目已从单 HTML 工具升级为“轻量 FastAPI + SQLite + 原生 HTML/ECharts”的经营分析平台底座。当前不引入微服务、重型 BI、复杂权限系统或 AI 报告生成，优先保证数据准确、口径统一、多人访问一致和 Ubuntu 可部署。

## 目标数据流

Excel 基表 → 后端导入 → 字段转换与校验 → SQLite 聚合表 → 服务端指标与查询 API → 前端 ECharts 展示。

## 分层职责

| 层级 | 目录 | 职责 |
|---|---|---|
| API 层 | `backend/api/` | 统一输出 `/api/kpi`、`/api/platform-trend`、`/api/org-analysis`、`/api/team-analysis`、`/api/product-analysis`、`/api/targets` |
| 服务层 | `backend/services/` | 数据转换、查询组装、缓存入口、统一响应格式 |
| 指标层 | `backend/metrics/` | 达成率、同比、环比、活动率、人均保费、人均产能等公式 |
| 校验层 | `backend/validators/` | 空值、重复、负数、月份、机构、业务线、目标 payload 校验 |
| 数据层 | `backend/database.py`、`backend/database/` | SQLite 表结构、索引、目标行表、迁移脚本 |
| 前端层 | `经营分析模板.html`、`frontend/src/` | 当前生产页面保留，新增模块化迁移目录和统一配置 |
| 部署层 | `deploy/` | Nginx、systemd、docker-compose、部署脚本 |

## 当前边界

1. 生产页面仍以 `经营分析模板.html` 为入口，保持深色 UI 和现有交互。
2. `frontend/` 目录已建立模块化迁移骨架，后续按模块逐步迁移，不做一次性大重写。
3. SQLite 当前满足 100+ 只读访问和低频导入目标维护，代码结构保留 PostgreSQL 迁移空间。
