# 系统架构说明

## 核心结论

本项目已从单 HTML 工具升级为“轻量 FastAPI + SQLite + 原生 HTML/ECharts”的经营分析平台底座。当前不引入微服务、重型 BI、复杂权限系统或 AI 报告生成，优先保证数据准确、口径统一、多人访问一致和 Ubuntu 可部署。

## 目标数据流

Excel 基表 → 后端导入 → 字段转换与校验 → SQLite 聚合表 → 服务端指标与查询 API → 前端 ECharts 展示。

## 分层职责

| 层级 | 目录 | 职责 |
|---|---|---|
| API 层 | `backend/api/` | 统一输出 `/api/platform-data`、`/api/kpi`、`/api/platform-trend`、`/api/org-analysis`、`/api/team-analysis`、`/api/product-analysis`、`/api/targets`；旧接口集中在 `backend/api/legacy.py` |
| 服务层 | `backend/services/` | 数据转换、查询组装、缓存入口、统一响应格式、导入安全写入 |
| 指标层 | `backend/metrics/` | 达成率、同比、环比、活动率、人均保费、人均产能等公式 |
| 校验层 | `backend/validators/` | 空值、重复、负数、月份、机构、业务线、目标 payload 校验 |
| 数据层 | `backend/db/` | SQLite 连接、表结构、索引、聚合表查询、目标行表 |
| 前端层 | `经营分析模板.html`、`js/api-client.js` | 当前生产页面入口，保留原生 HTML/JS/ECharts 展示；通用 API 客户端已拆出 |
| 部署层 | `deploy/` | Nginx、systemd、部署脚本、目标数据恢复脚本 |

## 当前边界

1. 生产页面仍以 `经营分析模板.html` 为入口，保持深色 UI 和现有交互。
2. 当前 FastAPI 默认运行库为 `backend/business_data.db`；根目录 `business_data.db` 属于历史/备份库，不作为服务默认读写对象，且不提交 Git。
3. 新旧 API 暂时并存：生产页面读取链路已优先调用 `backend/api/` 统一响应接口；`/api/data/{year}`、`/api/kpi/{year}`、`/api/product/{year}` 等兼容接口仅为历史入口和外部书签保留。
4. SQLite 当前满足 100+ 只读访问和低频导入目标维护，代码结构保留 PostgreSQL 迁移空间。
