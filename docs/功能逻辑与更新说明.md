# 经营分析看板功能逻辑与更新说明

更新日期：2026-05-08

## 一、核心结论

本次修正聚焦两个高风险口径问题：一是机构维度无表格展示，二是业务平台趋势在季度、月度切换时未按日累计趋势同口径展示。修正后，项目形成“Excel 源数据解析 - 后端聚合表 - API 返回 - 前端展示”的闭环，机构维度、季度/月度趋势、上一年度同期对比均以统一聚合口径驱动。

## 二、项目功能边界

1. KPI 概览：展示整体、经代、转型业务及 OTO、证保、蚁桥的期交保费、价值保费、达成率、同比等核心指标。
2. 机构维度：按机构和业务模式展示期交保费、价值保费、10 年期产品、商保年金、保障类产品的目标、达成、达成率及同比。
3. 业务平台趋势：支持年度、季度、月度切换，支持期交保费、规模保费、折算保费切换，支持业务线和机构筛选。
4. 产品结构：按产品设计分类、缴费年限、保障年限、是否商保年金等维度展示保费和件数结构。
5. 队伍分析：展示人力、活动人力、产能等队伍经营指标。
6. 目标管理：支持年度、季度、月度目标维护，并与实际达成口径进行匹配。

## 三、数据逻辑

### 1. 数据入口

项目当前主要使用四类本地 Excel 数据：

1. 经营分析业绩基表：提供 OTO、证保、蚁桥的保费、产品、机构等明细。
2. 经代业绩分析：提供经代业务保费数据。
3. 人力基表：提供队伍规模、活动人力等人力数据。
4. 经营分析价值基表：提供价值保费数据。

### 2. 聚合表设计

本次补齐后，数据库聚合层包括：

1. `agg_performance`：转型业务按年、月、业务模式聚合。
2. `agg_jingdai`：经代按年、月聚合。
3. `agg_hr_data`：人力按年、月、业务模式聚合。
4. `agg_value_data`：价值保费按年、月、业务模式聚合。
5. `agg_product_structure`：产品结构维度聚合。
6. `agg_daily_performance`：业务模式日维度保费聚合。
7. `agg_org_daily_performance`：机构、业务模式、日维度保费聚合。
8. `agg_org_performance`：机构、业务模式、月维度业绩聚合。
9. `agg_org_value`：机构、业务模式、月维度价值聚合。

### 3. 口径规则

1. 年度口径：按当年全部月份累计。
2. 季度口径：按所选季度内 3 个月累计，Q1 为 1-3 月、Q2 为 4-6 月、Q3 为 7-9 月、Q4 为 10-12 月。
3. 月度口径：按所选月份累计。
4. 同比口径：上一年度同期间、同业务线、同机构、同指标口径计算。
5. 机构筛选口径：OTO、证保、蚁桥按所选机构过滤；经代当前无机构维度，仍按经代整体口径纳入业务平台趋势。
6. 日累计口径：如源数据含日期字段，按日期逐日累计；如源数据仅含年月字段，则以该月 1 日作为当前可用最细粒度，展示为单点累计。

## 四、本次问题诊断

### 1. 机构维度未展示表格

原因：

1. 前端已存在机构表格渲染函数，但本地数据库缺少 `agg_org_performance`、`agg_org_value`、`agg_daily_performance` 等新聚合表数据。
2. `backend/rebuild_from_excels.py` 只重建了旧表，未写入机构业绩、机构价值和日累计聚合数据。
3. 机构维度前端采用固定十家机构清单，展示范围应与业务管理口径保持一致。

影响：

1. 机构维度无法形成表格展示。
2. 即使接口恢复，也可能因机构清单不完整导致部分机构缺失。
3. 季度、月度维度下存在“目标按期间、实际按全年”的口径错配风险。

修正：

1. 补齐机构业绩、机构价值、日累计、机构日累计聚合。
2. 重建脚本同步写入新增聚合表。
3. 机构表格改为年度、季度、月度分别取同期间实际值和上年同期值。
4. 机构维度和平台趋势机构筛选维持现有十家机构口径。
5. 保留非零负数行，避免冲正、退保调整等业务数据被误过滤。

### 2. 业务平台趋势季度、月度展示错误

原因：

1. 季度视图原逻辑以月为横坐标，未按日累计趋势展示。
2. 月度视图在机构筛选时退化为柱状图，未满足日累计趋势口径。
3. 上年同期趋势线在不同筛选条件下未完全使用同口径日累计。

影响：

1. 季度、月度切换时横坐标粒度不一致。
2. 机构筛选后当前年和上年同期不可直接比较。
3. 管理层查看趋势时容易将月累计点误读为日累计过程。

修正：

1. 季度视图改为季度内每日累计横坐标，例如 `4月1日、5月1日、6月1日`；如源数据具备真实日期，将自动展开为自然日序列。
2. 月度视图统一为当月日累计横坐标。
3. 当前年度与上一年度同口径趋势线统一由同一套日累计函数生成。
4. 机构筛选时优先使用 `agg_org_daily_performance` 形成机构日累计；无日字段时退化为源数据可支持的最细粒度单点累计，不再切换为柱状图。

## 五、更新清单

1. `backend/aggregator.py`
   - 新增 `aggregate_org_daily_performance`。
   - 支持按年、月、日、机构、业务模式聚合期交、规模、折算保费。

2. `backend/database.py`
   - 新增 `agg_org_daily_performance` 表。
   - `get_platform_data` 返回机构日累计数据。
   - `get_org_kpi_data` 返回年度、季度、月度三层结构及上年同期同口径数据。

3. `backend/main.py`
   - 上传接口同步写入机构日累计聚合表。
   - 年份识别范围纳入新增机构日累计数据。

4. `backend/rebuild_from_excels.py`
   - 本地重建脚本补齐机构业绩、机构价值、日累计、机构日累计写入。

5. `经营分析模板.html`
   - 机构表格按年度、季度、月度同口径展示实际值、目标值和同比。
   - 业务平台趋势季度、月度统一为日累计折线。
   - 上一年度同期线按同口径生成。
   - 机构清单维持现有十家机构。
   - 非零负数经营数据不再被过滤。

## 六、验证结果

本次本地重建后，数据库聚合结果如下：

1. `agg_org_performance`：483 条，其中 2026 年 74 条。
2. `agg_org_value`：74 条，其中 2026 年 74 条。
3. `agg_daily_performance`：87 条，其中 2026 年 15 条。
4. `agg_org_daily_performance`：483 条，其中 2026 年 74 条。
5. `get_org_kpi_data(2026)` 已返回 10 家机构列表和年度、季度、月度分层数据。
6. `get_platform_data(2026)` 已返回 `org_daily_performance`。

说明：当前源业绩基表未识别到真实日字段，因此日累计聚合以“年月”回退到每月 1 日。后续如源数据增加“日期、出单日期、投保日期、承保日期”任一字段，系统会自动展示真实日累计曲线。

## 七、后续建议

1. 数据源治理：在业绩基表中保留真实出单日期字段，作为季度、月度日累计趋势的基础字段。
2. 机构清单治理：建议后续由接口返回机构清单动态生成筛选项，避免新增机构后前端遗漏。
3. 口径治理：目标配置应补齐十家机构的年度、季度、月度目标，否则实际达成可展示但目标、达成率为空。
4. 校验机制：每次导入后应校验聚合表行数、机构清单、月份覆盖、上年同期覆盖，避免页面展示正常但口径缺失。
# 2026-05-09 应用级重构与修复

## 核心结论

本次调整在保留现有页面风格和业务主线基础上，补齐了轻量平台化底座：统一指标中心、统一业务线配置、统一 API 层、目标服务端行表、数据校验、日志、部署文件和测试用例。前端生产入口仍为 `经营分析模板.html`，避免一次性重写造成展示风险。

## 重点修复

1. 平台趋势月份比较改为 `normalizeMonth`/数字化比较。
2. 经代日累计先判断 `daily_performance` 是否已包含经代，避免与 `jingdai_daily` 重复叠加。
3. 经代在机构筛选下按整体口径展示，并在图表下方提示口径。
4. 无真实日数据时不再用月度值生成伪曲线。
5. 目标数据继续兼容现有 JSON 配置，同时同步写入 `target_values` 行表，支撑多人访问和后续查询扩展。

# 2026-05-09 业务平台趋势 & 队伍分析 bug 修复

## 核心结论

本次修复针对两个实际运行 bug，不新增功能、不重构页面风格。核心改动：补齐队伍分析模块的上年数据加载、修正平台趋势日累计在无数据时的渲染防御、统一图表状态管理。

## Bug 1: 经代平台趋势 季度/月度无趋势线

**根因分析**：
- `buildDailyTrendOption` 在当年日累计数据为空时直接返回空白图表（early return），导致即使上年同期数据可用也无法展示。
- 后端 `get_platform_trend` 已正确生成 `daily` 字段（periodType=month 且 periodValue 存在时），前端 `getMonthDailyCumulative` 也能正确读取 `jingdai_daily`，但缺少防御性回退。

**修复内容**：
1. `buildDailyTrendOption`：不再在当年数据为空时立即返回空图表，先同时计算当年和上年数据，仅当两者皆空时才展示空状态。
2. 当年数据为空但上年数据存在时，以上年数据为主展示，确保上年同期趋势线可正常渲染。

## Bug 2: 队伍分析默认不显示去年同期趋势线

**根因分析**：
- `loadYearFromApi` 在加载上年数据后，仅调用 `convertApiToPlatformMock` 填充 `platformMock`，未调用 `convertApiToTeamMock` 填充 `teamMock`。
- 因此 `getTeamOption` 检测 `teamMock[prevYear]` 为 false，不添加上年趋势线。
- 当用户手动切换到上年（如 2025）再切回 2026 时，`switchTeamYear` 触发的 `loadYearFromApi(2025)` 已将 2025 数据写入 `teamMock`，`teamMock[2025]` 残留使得切回后上年趋势线出现。

**修复内容**：
1. `loadYearFromApi`：加载上年数据后同时调用 `convertApiToTeamMock` 并 `Object.assign(teamMock, prevTM)`，确保初始化时即加载上年队伍数据。

## 图表状态管理修复

- 所有图表切换函数（`switchYear`, `switchTimeDim`, `switchSubPeriod`, `switchPremiumType`, `toggleSeries`, `toggleOrg`, `switchTeamYear`, `switchTeamMetric`, `switchTeamDim`, `switchTeamQuarter`, `toggleTeamSeries`）均在 `setOption` 前调用 `chart.clear()` 清除旧 series，防止年份切换后旧趋势线残留。

## 测试补充

- `normalize_month` 扩展：覆盖 `20260401`, `2026-04-01`, `2026/04/01`, `4`, `"04"` 等 7 种输入格式
- 经代日累计链路：有日期字段时生成日累计、month 参数缺失时 periodValue 驱动 daily、daily_performance 含/不含经代的去重逻辑
- 平台趋势：经代月度/季度趋势包含当年和上年线
- 队伍分析：上年数据可正常加载
