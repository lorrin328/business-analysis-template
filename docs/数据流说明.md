# 数据流说明

## 当前数据流

1. 用户上传业绩、经代、人力、价值 Excel。
2. Web 上传入口 `/api/upload` 与本地重建脚本 `backend/rebuild_from_excels.py` 统一调用 `backend/services/excel_pipeline.py`。
3. `excel_pipeline.py` 按源文件类型调用 `backend/etl/` 下的解析与聚合函数：
   - `etl/aggregates/performance.py`：转型业绩、日级累计、机构业绩、交期、长险期交；
   - `etl/aggregates/jingdai.py`：经代业绩、经代日级累计、经代交期、经代长险期交；
   - `etl/aggregates/hr.py`：人力、机构人力、活动人力回填；
   - `etl/aggregates/value.py`：价值保费、机构价值；
   - `etl/aggregates/product.py`：产品结构聚合。
4. `backend/services/data_transform.py` 统一月份、数字、机构、业务线等字段转换规则。
5. `backend/validators/` 对导入结果做基础校验，`backend/services/import_safety.py` 负责增量写入原始表时的期间删除和列结构兼容。
6. 聚合表和原始明细表写入 `backend/business_data.db`，可通过 `BUSINESS_ANALYSIS_DB` 覆盖运行库路径。
7. 前端通过 FastAPI 接口读取标准数据并渲染图表。

## 导入入口与维护边界

| 场景 | 入口 | 写库方式 | 维护重点 |
|---|---|---|---|
| Web 页面上传 Excel | `backend/main.py` 的 `/api/upload` | 增量写入，保留未覆盖期间数据 | 保留重复文件跳过、部分失败拦截、导入历史和操作日志 |
| 本地/部署全量重建 | `backend/rebuild_from_excels.py` | 全量替换聚合表和原始表 | 根目录必须存在四类源 Excel，适合部署初始化或源文件整体刷新 |
| 后续新增聚合表 | `backend/services/excel_pipeline.py` + `backend/etl/aggregates/` | 由 pipeline 统一调度 | 优先更新 `AGGREGATE_TABLE_ORDER`、解析聚合函数和写库结果，不要在上传入口单独实现 |

## 原始表

| 表 | 来源 | 用途 |
|---|---|---|
| `performance` | 转型业绩 Excel | 产品结构、转型产品分类标识、原始明细复核 |
| `jingdai` | 经代业绩 Excel | 经代产品配置提取、产品结构、原始明细复核 |
| `hr_data` | 人力 Excel | 队伍结构与产能分析、活动人力复核 |
| `value_data` | 价值 Excel | 价值保费复核 |

## 数据校验

严重错误阻断导入：必填字段缺失、月份异常、负数金额、聚合唯一键重复。一般警告允许导入但需记录：机构未登记、业务线未登记、转型与经代日级截止日不一致。

## 扩展新模块的建议顺序

1. 先确认模块使用原始表、聚合表还是目标配置表，避免同一指标出现双口径。
2. 若需要新增导入字段或聚合表，优先在 `backend/etl/aggregates/` 增加解析/聚合逻辑，再接入 `backend/services/excel_pipeline.py`。
3. 若只新增查询或展示模块，优先在 `backend/db/repositories/` 增加查询函数，在 `backend/api/` 暴露接口，并通过 `services.response.response_meta()` 统一响应说明。
4. 若直接读取原始表日期字段，优先复用 `backend/services/raw_table_reader.py`，不要在 repository 内重新拼接日期 SQL。
5. 完成后同步更新 `docs/ai-context/WORKLOG.md`、`DECISIONS.md`、`TODO.md` 或 `DATA_DICTIONARY.md` 中对应内容。
