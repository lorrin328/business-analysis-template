## 阅读顺序建议

**新接手项目者**：
1. 根目录 `README.md`（项目背景与当前实现）
2. `docs/需求/2026-04-26_模块化重构总体方案.md`（架构与演进路线）
3. `docs/数据规范/保单数据规范.md`、`docs/数据规范/人力数据规范.md`（数据源契约）
4. 各模块需求（按需）

**评审者**：先读「主方案」第 1-3 章（背景、目标、范围）+ 第 6 章（实施路线图）+ 第 9 章（开放问题）。

## 文档生命周期

| 阶段 | 状态 | 含义 |
|------|------|------|
| `[草稿]` | 撰写中 | 仍可大改 |
| `[评审中]` | 待用户确认 | 等待用户 review |
| `[已确认]` | 范围锁定 | 进入实施 |
| `[已实施]` | 代码完成 | 仍可作为历史参考 |
| `[已废弃]` | 不再适用 | 保留作为决策记录 |

每个文档头部应明确状态。状态变更时在「修订记录」表追加一条。

---

## 文档命名约定

- 主方案与跨模块文档：`docs/需求/YYYY-MM-DD_主题.md`
- 模块独立文档：`docs/需求/模块NN_主题.md`（编号稳定，便于交叉引用）
- 数据规范：`docs/数据规范/{保单,人力,...}数据规范.md`
- 修订记录：唯一 `docs/修订记录/CHANGELOG.md`，时间倒序

---

## 与 GitHub 的同步规则

1. 所有 docs 变更与代码变更**必须在同一个 commit 或紧邻的 commit 中**推送
2. Commit message 引用文档路径：`docs: update 模块03_队伍规模 (refs docs/需求/模块03_队伍规模.md)`
3. 推送前自检：
   - 文档头部状态字段是否更新？
   - CHANGELOG 是否追加摘要条目？
   - README 「修改历史」是否同步？
4. 推送后 GitHub URL：`https://github.com/lorrin328/business-analysis-template/tree/master/docs`
