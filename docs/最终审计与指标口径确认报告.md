# 经营分析看板最终审计与指标口径确认报告

## 一、审计结论

当前版本具备作为后续功能扩展底座的基本条件：FastAPI + SQLite + 原生 HTML/ECharts 的主链路清晰，后端 API、聚合、服务、配置、指标公式已有分层，自动化测试当前全部通过。

但在正式冻结底座前，仍有三类事项需要确认或治理：

1. **时间截止口径必须统一**：期交保费支持日级截止，经代与转型必须按同一截止日计算；长险期交、价值、产品类指标当前为月级聚合，导入批次必须保证转型与经代处于同一统计截止月。
2. **文档存在新旧口径冲突**：`docs/指标口径说明.md`较贴近当前实现；`docs/指标口径手册.md`及部分需求文档仍保留旧的 `fact_premium/fact_headcount` 设计口径，与当前 SQLite 聚合表实现不一致。
3. **前端仍保留本地兜底数据与非运行 JS 模块**：这保证 API 慢或短时不可用时页面不空白，但若服务端异常，页面可能显示本地兜底数据；`js/` 下除 `api-client.js` 外多为迁移参考文件，不能作为生产逻辑依据。

## 二、架构审计

### 1. 当前生产运行链路

Excel 基表 → 后端导入校验 → SQLite 原始表与聚合表 → FastAPI 查询接口 → `经营分析模板.html` + `js/api-client.js` 展示。

### 2. 分层评价

| 层级 | 当前状态 | 审计判断 |
|---|---|---|
| 前端入口 | `经营分析模板.html` 为生产页面，`js/api-client.js` 为实际加载的共享 API 客户端 | 可运行，但单文件过大，后续新增模块前建议拆分 |
| API 层 | `backend/api/` 按 kpi、trend、org、team、product、targets、payment_period、product_config 拆分 | 结构清晰 |
| 数据层 | `backend/db/schema.py`、`backend/db/repositories/` 管理表结构与查询 | 基本合理，SQLite 能覆盖当前低频导入与多人只读场景 |
| ETL 层 | `backend/etl/aggregates/` 分业务聚合 | 业务聚合已收敛；导入阶段已增加转型/经代日级截止一致性校验 |
| 指标层 | `backend/metrics/formulas.py`、`backend/metrics/business_rules.py` | 通用公式已集中，具体 KPI 口径仍有一部分写在 repository 与前端展示逻辑中 |
| 配置层 | `backend/config/` 管理业务线、机构、指标配置 | 已具备扩展基础 |
| 文档层 | 架构说明较新，口径手册和需求文档存在旧设计痕迹 | 需统一权威文档 |

### 3. 架构风险

| 风险 | 影响 | 建议 |
|---|---|---|
| 单 HTML 文件承载大量展示逻辑 | 后续新增模块时容易引入联动错误 | 冻结当前功能后，按 KPI、机构、趋势、产品、目标、参数设置拆分模块 |
| 本地兜底数据仍在生产页面内 | API 异常时可能误以为是服务端真实数据 | 增加明显“服务端数据/本地兜底”状态标识 |
| `js/` 中存在非运行参考模块 | 维护者可能误改无效文件，或复制旧逻辑 | 明确归档或删除非运行模块，仅保留实际加载文件 |
| 新旧目标存储并存 | `target_config` JSON 与 `target_values` 行表可能被误解为双数据源 | 明确一个写入主源，另一个作为查询索引或后续迁移目标 |
| 文档引用旧事实表 | 新模块开发可能按旧表设计 | 将旧需求文档标为历史设计，新增一份当前实现口径手册 |

## 三、指标与计算口径确认表

### 1. 通用公式

| 指标 | 公式 | 不可计算规则 | 当前实现 |
|---|---|---|---|
| 达成率 | 实绩 / 目标 | 目标为空或 0 时不可计算 | 后端公式存在，前端展示中部分场景仍本地计算 |
| 同比 | 本期 / 去年同期 - 1 | 去年同期为空或 0 时不可计算 | 期交保费优先日级同口径；长险期交为月级 |
| 环比 | 本期 / 上期 - 1 | 上期为空或 0 时不可计算 | 通用公式已定义，当前不是核心展示重点 |
| 序时进度 | 已过时间 / 总周期 | 周期参数缺失时不可计算 | 已定义，部分展示未深度使用 |
| 进度偏差 | 达成率 - 序时进度 | 任一项不可计算则为空 | 已定义 |
| 活动率 | 活动人力 / 在职人力 | 在职人力为空或 0 | 当前长险活动率用月均在职口径 |
| 人均保费 | 期交保费 / 月均在职人力 | 月均在职为空或 0 | 当前 KPI 为转型业务口径 |
| 人均产能 | 期交保费 / 月均活动人力 | 月均活动人力为空或 0 | 后端公式存在，展示依模块使用 |

### 2. KPI 概览口径

| KPI | 分子 | 分母/目标 | 时间口径 | 数据源 | 审计判断 |
|---|---|---|---|---|---|
| 期交保费达成率 | 经代 + OTO + 证保 + 蚁桥期交保费 | 期交保费年度目标 | 优先日级累计；转型与经代日表不一致时统一取共同最早截止日；无完整日表时回退共同月级 | `agg_daily_performance`、`agg_jingdai_daily`、`agg_performance`、`agg_jingdai` | 公式正确；需展示统一截止日 |
| 价值达成率 | OTO + 证保 + 蚁桥价值保费 | 价值年度目标 | 月级累计 | `agg_value_data` | 公式正确；经代无价值口径 |
| 长险活动率 | 长险活动人力 | 月均在职人力 | 月级，取当前统计月 | `agg_hr_data` | 口径清晰；同比按百分点差展示 |
| 商保年金达成率 | 参数设置标识为商保年金的产品期交保费，含转型与经代 | 商保年金目标 | 月级累计 | `agg_org_performance.product_annuity`、`agg_jingdai.product_annuity`、`product_config` | 逻辑正确；依赖参数设置后重新计算 |
| 保障类产品达成率 | 参数设置标识为保障类的产品期交保费，含转型与经代 | 保障类产品目标 | 月级累计 | `agg_org_performance.product_protection`、`agg_jingdai.product_protection`、`product_config` | 逻辑正确；本地库当前保障类实绩为 0，需以生产库配置为准 |
| 10年期产品达成率 | 转型缴费年限 ≥10 + 经代交期分类“10年及以上” | 10年期产品目标 | 月级累计 | `agg_org_performance.product_10year`、`agg_payment_period` | 已包含经代；口径正确 |
| 长险期交达成率 | 长险期交保费，含转型与经代 | 期交保费年度目标 | 月级累计 | `agg_longterm_qj` | 分子逻辑正确；分母明确沿用期交保费目标 |
| 人均保费 | 转型业务月均期交保费 | 转型月均在职人力 | 期交按所选期间先转为月均，人力按期间月均 | `agg_daily_performance`、`agg_hr_data` | 可用；不含经代 |

### 3. 明细模块口径

| 模块 | 指标/维度 | 当前口径 | 数据源 | 审计判断 |
|---|---|---|---|---|
| 机构维度 | 期交、价值、10年期、商保年金、保障类 | 按机构 + 业务模式展示目标、实绩、达成率 | `agg_org_performance`、`agg_org_value`、`target_config` | 可用；经代不进入机构维度 |
| 平台趋势 | 期交/规保/折算保费累计趋势 | 年度/季度/月度，月度和季度支持真实日累计 | `agg_daily_performance`、`agg_jingdai_daily`、`agg_performance`、`agg_jingdai` | 已避免伪造曲线；需显示无日数据时的空状态 |
| 产品结构 | 产品设计分类或产品明细结构 | 设计分类走聚合表，产品明细走原始表查询 | `agg_product_structure`、`performance`、`jingdai` | 可用；原始表字段变动敏感 |
| 交期结构 | 趸交、短期险、3年交、5年交、10年及以上 | 转型按缴费年限/长短险，经代按当前缴别大类/缴费年限 | `agg_payment_period` | 口径清晰；经代件数当前为 0，因为源表无件数列 |
| 队伍分析 | 在职、活动、人均相关 | 仅转型业务，基于人力基表 | `agg_hr_data`、`agg_org_hr_data` | 可用；经代无队伍人力口径 |
| 参数设置 | 商保年金/保障类产品标识 | 按 `business_type + product_code` 唯一配置，经代产品也可配置 | `product_config` | 当前设计正确；保存后会重算相关聚合 |
| 目标设置 | 年/季/月、整体/业务线/机构目标 | JSON payload 入库，同时展平到行表 | `target_config`、`target_values` | 可用；需明确权威主源 |

### 2. 仍需治理的问题

| 等级 | 问题 | 影响 | 建议 |
|---|---|---|---|
| 高 | 月级指标无法精确到日，依赖导入批次本身同口径 | 横向比较时可能产生截止精度差异 | 日级截止不一致已在导入阶段拦截；月级指标需继续展示截止口径 |
| 高 | `docs/指标口径手册.md` 与当前实现冲突 | 后续开发可能按旧表和旧口径实现 | 将其标记为历史文档或重写为当前实现口径手册 |
| 中 | 生产页面内置本地兜底数据 | 服务端异常时可能误读为真实数据 | 增加显著数据来源状态，兜底状态下禁用达成率或加警示 |
| 中 | 非运行 JS 模块保留旧逻辑 | 维护者可能误改无效代码 | 迁移到 `archive/` 或删除，只保留实际运行脚本 |
| 中 | 默认年份随系统日期滚动 | 到 2027 年若尚未导入 2027 数据，可能默认打开空数据 | 默认取数据库最大年份或服务端返回可用年份 |
| 中 | 产品结构明细直接查原始表 | 原始 Excel 字段变化时容易失效 | 后续改为导入时聚合更多产品维度 |
| 低 | 文档仍出现旧默认 Token 示例 | 运维文档可能误导 | 更新 Docker/部署文档，避免明文默认口令 |

## 六、冻结底座前建议确认的问题

1. **长险期交达成率分母**：已确认继续使用期交保费目标，不另设独立长险期交目标。
2. **截止日策略**：已确认必须保证转型与经代为同一截止日。当前执行规则为日级指标按共同最早截止日计算，并在后续导入中拒绝转型/经代日级截止日不一致的数据文件。
3. **保障类产品配置**：本地库保障类实绩为 0，说明本地配置未打标或未重算；生产库如已配置，应以生产库抽查为准。
4. **目标数据主源**：指定 `target_config` JSON 为当前权威主源；`target_values` 作为展平索引表和后续查询优化表，不作为并行录入主源。

## 七、结论

当前版本不是“明显错误不可用”的状态，主链路已经稳定，测试通过，核心 KPI 计算路径可追溯。但如果把它作为后续功能扩展的长期底座，建议先完成两件事：

1. 将本报告中的指标口径确认表作为当前权威口径，清理或降级旧口径文档。
2. 对“数据截止日/截止精度”做产品级约束：日级指标按共同截止日计算，导入时校验、API 返回、前端展示三处保持一致。

完成上述治理后，再新增模块风险会明显降低。
