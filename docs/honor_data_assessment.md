# 星钻联盟现有数据适配评估报告

## 一、结论

当前平台已导入的 `performance` 与 `hr_data` 字段，可以支撑“个人星钻 MVP”的核心计算：转型 OTO、证保个人月度达标、获钻、扣减、证保保号、月末非在职清零、会员等级、新星人力、人员明细和异常清单。

本阶段不建议新增星钻专用 Excel 上传。经代和蚁桥（网服）均不涉及星钻，不属于字段缺口；后续团队星钻也只归集 OTO、证保，团队内网服/蚁桥人员及业绩均不计入。

## 八、是否需要新增上传

本阶段不需要新增星钻专用上传。

1. `performance` 已包含 OTO、证保个人星钻所需字段。
2. `hr_data` 已包含 OTO、证保人员月度状态和入职字段。
3. 经代和蚁桥（网服）均不涉及星钻，不需要为其新增星钻上传或补字段。

## 九、当前最小实现范围

可先实现以下 MVP：

- 转型 OTO、证保个人月度星钻；
- 月度达标、获钻、扣减、证保保号；
- 月末非在职清零；
- 当前累计钻石和会员等级；
- 新星人力识别；
- 人员汇总、机构汇总、异常清单和 Excel 导出。

暂不纳入 MVP 的规则：

- 经代和蚁桥（网服）星钻：不涉及星钻；
- 主管/经理团队星钻：字段具备初步条件，但需防止重复归集，且必须排除团队内网服/蚁桥人员和业绩；
- 证保季度通算：需完整自然季度和缺月异常校验，建议 Phase 3 实施；
- 季度奖励：可以测算，但必须标识“预计/测算奖励”，不得作为最终发放金额。

## 十一、后续开发边界

可以进入个人星钻 MVP，但仅限 OTO、证保；不得把经代、蚁桥（网服）纳入个人或团队星钻，不得把团队星钻混入个人星钻，不得在缺字段时静默计算。所有不确定规则必须进入 `honor_exceptions`。
