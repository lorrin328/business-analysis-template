### 4.2 模块化方案：ESM + 发布期合并

**用户决策（2026-04-26）**：采用 ES Module + 发布期合并。

**开发期工作流**：

```bash
# 1. 启动本地 http server（ESM 在 file:// 下受 CORS 限制）
cd business-analysis-template
python3 -m http.server 8000

# 2. 浏览器打开 http://localhost:8000/经营分析模板.html
# 3. 改 js/ 下任意文件，刷新即生效
```

**发布期工作流**：

```bash
# build.sh 把 js/* 全部内联到 HTML，输出单文件
./build.sh

# 输出：dist/经营分析模板.html（双击即用，含全部依赖）
```

**build.sh 简易实现**：bash + `sed` 把 `<script type="module" src="./js/app.js">` 替换为 `<script>` + 文件内容拼接。约 50 行 shell。

**两种形态对比**：

| 形态 | 用途 | 限制 |
|------|------|------|
| 开发版（多文件 ESM） | 编辑、调试、测试 | 需要本地 http server |
| 发布版（单文件 HTML） | 用户分发、双击使用 | 单文件，便于邮件传输 |

### 4.3 数据存储升级：sqlite-wasm + OPFS

**用户决策**：纳入 P2 阶段。

**关键差异**：

| 项 | sql.js（当前） | sqlite-wasm + OPFS（升级后） |
|----|----------------|------------------------------|
| 维护方 | 第三方 emscripten 移植 | SQLite 官方（2023 年发布） |
| 持久化 | 全库 export 到 IndexedDB | 直接写 OPFS 文件，按页操作 |
| 启动 | 加载完整 buffer 到内存 | 文件句柄打开，零拷贝 |
| 写入 | 每次需全量 export 回写 IDB | 增量 fsync，写一行刷一页 |
| 容量 | 实测 8-12 MB（保单年级） | 测试可至 1 GB+（多年多表） |
| 兼容 | 全浏览器 | Chrome 86+ / Edge 86+ / Safari 17+ / Firefox 111+（限制中） |

**API 兼容性**：sqlite-wasm 提供 `oo1` 高级 API，写法与 sql.js 接近。封装层 `js/core/db.js` 抹平差异，业务代码无感。

**降级策略**：在 OPFS 不可用时（旧 Safari），回退到 sqlite-wasm 内存模式 + IndexedDB 持久化（性能与 sql.js 相当）。

### 4.4 数据模型演进

#### 4.4.1 表分类

```
维度表 (dim_*) - 缓慢变化，多个事实表共享
├── dim_org           # 销售机构
├── dim_staff         # 人员档案（含层级、入职信息）
├── dim_product       # 产品（产品代码、名称、设计分类）
└── dim_period        # 时间维度（year/quarter/month）

事实表 (fact_*) - 业务发生记录
├── fact_premium      # 保单明细（已有）
├── fact_headcount    # 人力月度快照（新增，对应 N1AI 表）
└── fact_activity     # 活动率快照（可选，或动态计算）

元数据 (meta_*)
└── meta_schema_version  # 迁移版本号
```

#### 4.4.2 关键设计原则

1. **维度表与事实表分离**：减少冗余，便于多模块共享
2. **统一时间粒度**：所有事实表都含 `year + month` 字段，方便 JOIN
3. **金额一律存"分"**（INTEGER）：沿用现有原则
4. **可空字段保 NULL**：避免 `'未知'` 字符串污染聚合
5. **小计行导入时过滤**：所有 fact 表入库前剔除 `WHERE 维度 = '小计'` 行

#### 4.4.3 字段扩展机制

新增字段流程（目标：< 0.5 天）：

1. 在 `js/core/schema.js` 末尾追加迁移：
   ```js
   migrations.push({
     version: 5,
     up: db => db.exec("ALTER TABLE fact_premium ADD COLUMN new_field TEXT")
   });
   ```
2. 启动时 `runMigrations()` 自动检测当前 `meta_schema_version` 与最新版本，按序执行 up
3. 数据规范文档对应章节追加字段说明
4. CHANGELOG 记录

**不需要**：清空数据、重新导入、修改业务模块代码（除非该模块要用新字段）。

### 4.5 全局状态与模块通信

```
┌─────────────────────────────────────────────────┐
│              app.js (主入口)                      │
│  ┌──────────────────────────────────────────┐   │
│  │  filters.js: 全局筛选状态                  │   │
│  │  - 时间范围、机构、保费口径、基准年         │   │
│  │  - on('change', cb) / get() / set()      │   │
│  └─────────────┬────────────────────────────┘   │
│                │ subscribe                       │
│  ┌─────────────┴────────────────────────────┐   │
│  │  modules/* 各模块                         │   │
│  │  - 监听筛选变更 → 重查询 → 重渲染          │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

每个模块需实现：

```js
export default {
  id: 'platform-trend',
  label: '平台趋势',
  icon: '📈',
  mount(container, ctx) { /* 挂载 DOM */ },
  unmount() { /* 清理 */ },
  refresh() { /* 重查询、重渲染 */ }
};
```

`app.js` 维护当前激活模块，切 tab 时 unmount 旧的 + mount 新的。

## 5. 业务模块概览

详见各模块独立需求文档。**模块布局采用 Tab 切换**（用户 2026-04-26 决策），每个模块独立 Tab 页，不做单页滚动。

| # | 模块 | 数据源 | 关键 KPI | 状态 |
|---|------|--------|----------|------|
| 01 | 平台趋势 | fact_premium | 累计保费、同比增速；基准年=系统当前年份（自动滚动） | 已有，需迁移 |

| 03 | 队伍规模 | fact_headcount | 月末/月均在职人力；含「外勤管理职」聚合（主管+经理）；留存率延后 v2 | 待开发 |
| 04 | 长险活动率 | fact_premium ⋈ fact_headcount | 长险活动率 = 长险活动人力 / 月均在职人力 | 待开发 |
| 05 | 队伍产能 | fact_premium ⋈ fact_headcount | 三定义：人均保费（期交/月均在职）、人均产能（期交/月均活动人力）、人均件数（不含附加险/月均在职） | 待开发 |

**术语规范**（用户 2026-04-26 决策）：

- **在职人力**：当月有人力档案的所有员工（hc_end=1 或 hc_start=1）
- **活动人力**（取代「活跃人力」）：当月有保单（zhsf>0）的员工
- **长险活动人力**：当月有长险保单的员工（活动人力的子集）
- **外勤管理职**：主管 + 经理 聚合层级（具体职级边界待定，见 Q9）
- **基准年**：系统当前年份，自动滚动（2026 年用 2026 作基准；2027 年自动切到 2027）

## 附录 A：技术决策记录（ADR）

### ADR-001：选择 ESM + 发布期合并而非全局命名空间

**日期**：2026-04-26
**决策**：开发期 ES Module 多文件，发布期 build.sh 合并为单 HTML
**备选**：全局 `window.App` 命名空间 + 多个 `<script>` 标签
**理由**：
- ESM 提供清晰的依赖关系与按需加载
- 开发期受益于现代模块系统（命名导出、循环依赖检测）
- 发布期合并保留"双击 HTML"的核心特性
- 缺点（开发需 http server）通过 `python3 -m http.server` 一行解决

### ADR-002：选择 @sqlite.org/sqlite-wasm 而非继续 sql.js

**日期**：2026-04-26
**决策**：P2 阶段升级到 SQLite 官方 WASM 版本，启用 OPFS
**备选**：继续 sql.js + IndexedDB
**理由**：
- 官方维护，与 SQLite 主版本同步
- OPFS 真文件系统避免全库内存加载，支持 GB 级数据
- 多事实表 + JOIN 查询场景下性能优势明显
- 兼容性可接受（主流浏览器近 2 年版本均支持）
- 通过封装层 `db.js` 屏蔽差异，业务代码迁移成本低

### ADR-003：选择维度表 + 事实表星型模型

**日期**：2026-04-26
**决策**：引入 dim_* 与 fact_* 区分，多事实表共享维度
**备选**：保持单大宽表
**理由**：
- 多模块场景下，单大宽表会有大量 NULL 与冗余
- 星型模型降低存储、便于扩展（加新事实表不影响旧表）
- 维度变化时只需更新一处
- SQLite 索引优化对星型查询友好

### ADR-004：金额继续按"分"存 INTEGER

**日期**：2026-04-26（沿用既定规则）
**决策**：保留现有"元 × 100 → INTEGER"原则
**理由**：金融场景下浮点累计误差不可接受，已验证有效

### ADR-005：术语统一「活动人力」（取代「活跃人力」）

**日期**：2026-04-26
**决策**：全文档与代码统一使用「活动人力」一词；废弃「活跃人力」
**背景**：业务方惯用术语为「活动」（与"活动率"对齐），不用「活跃」
**影响范围**：模块 04、模块 05 文档；将来代码中变量命名（建议 `active_*` → `activity_*`，但 SQL 字段允许保留 `active_n` 以减少变更面）
**派生概念**：
- 活动人力 = 当月有保单（zhsf > 0）的员工
- 长险活动人力 = 当月有长险保单的员工（活动人力的子集）
- 活动率 = 长险活动人力 / 月均在职人力（默认口径）

### ADR-006：队伍产能口径收敛为三定义

**日期**：2026-04-26
**决策**：模块 05 仅实现 3 个产能定义（不再扩展）
**口径明细**：

| 指标 | 分子 | 分母 |
|------|------|------|
| 人均保费 | 期交保费 (qj_cents) | 月均在职人力 |
| 人均产能 | 期交保费 (qj_cents) | 月均活动人力 |
| 人均件数 | 件数（COUNT DISTINCT policy_no，**排除附加险**） | 月均在职人力 |

**取消的指标**：
- ❌ D5「合格人均长险」（用户：不需要）
- ❌ D6「司龄校准产能」（v1 不做）
- ❌「留存率」（用户：先不体现，后续再定义）

**理由**：业务方明确给出口径，无需穷举 6 种定义；先把核心的 3 个做好。

### ADR-007：基准年自动滚动

**日期**：2026-04-26
**决策**：基准年取自 `new Date().getFullYear()`，不硬编码、不允许用户在 UI 选择
**背景**：原始硬编码 `2024` 已多次成为 bug 来源；用户反复确认希望自动滚动
**实现**：在 `js/core/utils.js` 暴露 `getBaseYear()`；各模块统一调用
**注意**：测试时需 mock 系统日期；CHANGELOG 中需提示"每年元旦后第一次打开会自动切换基准年"

### ADR-008：外勤管理职聚合层级

**日期**：2026-04-26
**决策**：在维度模型中引入「外勤管理职」聚合层级，主管 + 经理 合并显示
**实现**：dim_staff 新增 `mgmt_level` 派生列（值域：'外勤管理职' | '一线营销员' | NULL）
**待确认**：具体职级边界由用户在 P3 启动前提供（见 Q9）
**回退**：若用户未及时提供，先以「supervisor_id 或 manager_id 非空」作为临时标记（来自保单表反推）
