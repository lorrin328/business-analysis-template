## 三、问题清单

### P0-1 生产服务器可匿名下载数据库、运行配置和仓库文件

证据：

- `deploy/nginx.conf:5,30-31` 把 `/opt/business-analysis` 作为站点根目录，并对任意现存路径执行 `try_files $uri`。
- `backend/main.py:337-343` 又把项目根目录整体挂到 `/static`，直连 FastAPI 端口时可访问项目内任意文件。
- 2026-07-14 对生产服务器只读 HEAD 验证：
  - `/backend/business_data.db` 返回 `200`，Content-Length `416706560`；
  - `/deploy/.admin_env` 返回 `200`；
  - `/deploy/.webhook_env` 返回 `200`；
  - `/.git/config`、`/targets_import.json`、`/js/seed-data.js` 均返回 `200`。

影响：经营原始明细、账号与 Session 哈希、目标、操作日志和部署密钥可能被绕过登录获取；已暴露的 webhook secret 不能再视为可信。

立即措施：

1. nginx 不得以项目根目录作为静态根；只允许白名单页面和 `js/` 资产，显式拒绝 `/backend`、`/deploy`、`/.git`、`/*.db`、`/*.env`、`/*.json`、`/*.xlsx`。
2. 删除 FastAPI 对项目根目录的 `/static` 挂载，改为专用静态资产目录。
3. 轮换 `.admin_env`、`.webhook_env`、`.ai_env` 中所有现用值，撤销现有登录 Session；检查 nginx 访问日志确认是否已被下载。
4. 处理完成前，不应把该服务暴露到更大网络范围。

### P0-2 `www-data` 可修改并以 root 执行部署脚本

证据：

- `deploy/deploy.sh:188-190` 为 `www-data` 配置免密执行 `/usr/bin/env bash /opt/business-analysis/deploy/deploy.sh`。
- `deploy/deploy.sh:199-200` 又把整个 `/opt/business-analysis` 递归改为 `www-data:www-data`，部署脚本本身可被应用账号修改。
- `deploy/webhook.service:7-13` 的 webhook 进程也以 `www-data` 运行。

影响：一旦应用进程、上传解析链或 webhook 被利用，攻击者可先修改 `deploy.sh`，再通过免密 sudo 获得 root。

建议：将 root 部署入口移到 `/usr/local/sbin` 等 root 所有、应用账号不可写的位置；代码目录只读，单独把数据库和日志目录授权给 `www-data`；sudoers 只允许固定 root-owned wrapper，不允许执行应用目录内可写脚本。

### P1-1 SQLite WAL 模式下直接 `cp` 备份不可靠

证据：`backend/db/connection.py:17` 启用 WAL；`deploy/deploy.sh:38-41` 在服务运行期间只复制主 `.db` 文件，未 checkpoint、未复制 `-wal`，也未使用 SQLite Backup API。

影响：备份可能缺少最近已提交事务，或恢复后与备份时点不一致；现有“备份成功”元数据不能证明可恢复。

建议：使用 SQLite Online Backup API/`sqlite3 .backup` 或停服务后执行 checkpoint 再复制，并增加自动恢复演练和关键表计数校验。

### P1-2 旧版只读接口绕过模块权限

证据：`backend/api/legacy.py:23-67` 的 `/api/data/{year}`、`/api/kpi/{year}`、`/api/product/{year}`、`/api/org-kpi/{year}`、`/api/targets/{year}` 仅依赖全局“已登录”中间件，没有对应 `require_permission()`。

影响：即使管理员关闭某用户的机构、产品或趋势权限，用户仍可直接调用旧接口取得数据。

建议：为旧接口补齐与新接口相同的权限依赖；发布一个版本后统计调用量，再下线旧接口。

### P1-3 v1.0.102“方案计算”实际读取 Excel 缓存结果，并非规则计算引擎

证据：

- `backend/scheme/calculator.py:392-397` 使用 `openpyxl(..., data_only=True)` 读取已缓存公式值，未从业绩、人力、参数重新计算政策规则。
- 新建含公式但未由 Excel 预计算的工作簿抽样结果：达标、奖励、最终奖励全部被读取为 `0`。
- `backend/scheme/calculator.py:324-346` 对缺少全部关键工作表仅生成高风险 warning；API 仍会写入 `status='success'` 的最新批次。
- 上传只有任意一个无关工作表的文件，解析结果为 `totalTeams=0`，但不会被拒绝。

影响：用户修改底层清单但未用 Excel 重新计算并保存时，系统可能展示旧值或零值；错误文件还能覆盖“最近一次成功批次”。“导入并测算”名称会造成业务误解。

建议：当前版本改名为“底稿结果导入与复核”，并阻止缺表、空明细、关键缓存值缺失的文件入库；正式上线前实现独立规则函数、数据源适配、逐月计算、差异对账和异常清单，不能把 Excel 缓存值视为结算结果。

### P1-4 无正式目标时仍使用硬编码目标计算经营结论

证据：

- `js/target-modal.js:34-72` 内置整套年度、季度和月度默认目标。
- `js/kpi-cards.js:28-31,148-177` 在浏览器端重复计算达成率，并据此输出“领先/落后、节奏可控”等经营判断。
- 本地 `target_config` 当前为 `categories=null`、`updated_by=test`；数据质量审计仍返回 `ok`。

影响：正式目标缺失时，页面仍可能基于示例目标产生看似正式的达成率和经营研判。

建议：生产环境目标缺失时必须显示“未配置正式目标”，禁止输出达成率、进度差和经营判断；指标计算统一后移到服务端，前端只渲染带口径和可计算状态的指标载荷。

### P1-5 生产鉴权基线仍偏弱

证据：生产 `/api/auth/config` 当前返回 `allowPublicRegistration=true`；nginx 仅监听 HTTP；登录无失败限流/锁定，密码规则仅要求不少于 8 位；`backend/api/auth_routes.py:199-213` 管理员重置密码后未撤销该用户旧 Session。

影响：账号枚举、弱密码尝试、明文链路窃听和旧 Session 延续风险仍存在。公开注册会让任何可访问服务器的人默认获得大部分只读经营数据权限。

建议：关闭公开注册或改审批制；启用 HTTPS；增加登录限流、失败锁定、密码强度和 Session 清理；重置密码时撤销目标用户全部 Session。

### P2-1 数据质量审计覆盖不完整，`ok` 不能代表全项目数据可信

`backend/services/data_quality_audit.py:80-126` 只对部分 HR、日级保费和长险聚合做重建比对，未覆盖月度总表、机构表、产品结构、交期、价值、目标、方案批次、导入来源完整性和机构范围外数据。当前本地库 `data_imports=0`、正式目标为空，脚本仍为 `ok`。

建议：把“聚合一致性”“源数据新鲜度”“正式目标完整性”“导入批次可追溯”“机构范围外记录”“方案批次有效性”拆成独立检查，并让健康接口区分 `service health` 与 `business readiness`。

### P2-2 截止日权威文档与当前代码存在冲突

当前 `backend/db/repositories/kpi.py:113-119` 按转型、经代各自真实截止日取数；但 `docs/指标口径说明.md:34,50` 和 `docs/最终审计与指标口径确认报告.md:60,98` 仍写“统一取共同最早截止日”。

建议：由业务确认唯一口径；代码、API meta、前端说明、指标手册、审计报告必须同步。未确认前，不应使用“同日口径”描述混合 KPI。

### P2-3 固定十家机构范围导致 KPI 与机构表不能自然勾稽

`backend/config/orgs.py:1-2` 固定十家机构，`backend/etl/aggregates/org.py:50-51` 会过滤范围外记录；平台 KPI 不按机构范围过滤。本地截至 `2026-07-01`，平台转型期交比机构日表合计多 `0.1895` 万，来源为“深圳”和“其他”。

这可能是有意的管理范围，但必须在页面和审计中明确“平台总量含范围外机构、机构表仅十家”，并对范围外数据设置异常提示，避免未来新机构被静默遗漏。

### P2-4 前端仍有较强全局状态和历史数据耦合

`js/platform-seed-data.js` 约 1.6 万行、346KB，并与 `js/seed-data.js` 一起对所有用户加载；其中含具体业务保费、件数和产品数据。即使生产运行时清空对象，文件内容仍已下发且可匿名下载。KPI、目标和图表仍依赖多个全局变量，回归测试以静态字符串断言为主。

建议：生产构建不包含真实/近真实兜底数据；开发 mock 独立打包且不部署；继续把指标计算和数据契约后移到服务端，前端增加基于 API fixture 的行为测试。

## 四、正向结论

- Excel 写库采用事务、操作锁、重复哈希和增量期间保护，能降低并发导入及误覆盖风险。
- 核心 SQL 大多参数化，动态标识符多数有白名单或引用保护。
- 密码使用 PBKDF2，Session 只存 token 哈希；生产默认注册策略在代码层是 opt-in。
- `asOf`、来源截止和口径 warning 已形成共享服务，荣誉体系已具备较完整的规则、批次、异常和真实底稿校准测试。
- 285 项自动化测试、数据质量脚本和预检均通过，说明当前代码基线稳定；但这些结果不能覆盖上述生产配置和业务口径风险。
