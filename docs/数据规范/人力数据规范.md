## 1. 文件命名

```
N1AI-人力基表_<导出日期>_<导出时间序号>.xlsx
例：N1AI-人力基表_20260421_1637302.xlsx
```

- 文件名仅作识别
- 同一文件可包含多个统计月份（实测：单文件含 4 个月）
- 文件应在 `.gitignore` 中保护（含员工姓名等敏感信息）

## 3. 关键约束与校验

### 3.1 ⚠️ 小计行过滤（必做）

实测文件含 4 行小计行（每月 1 条），特征：

```
销售机构名称 = '小计'
销售序列 = '小计'
职等 = '小计'
月初在职人力 = 大数值（如 560）
```

**导入逻辑**：

```sql
-- 任何维度等于 '小计' 即跳过
WHERE org = '小计' OR sales_seq = '小计' OR rank_grade = '小计'
   OR rank_level = '小计' OR biz_mode = '小计' OR tenure_band = '小计'
```

或在 transformRow 中：

```js
if ([org, sales_seq, rank_grade, rank_level, biz_mode, tenure_band]
    .some(v => v === '小计')) return null;  // 跳过
```

如果不过滤，将导致**人力被重复计 2 倍**。这是数据正确性的关键。

### 3.2 唯一性约束与增量导入策略（用户 2026-04-26 决策）

**逻辑主键**：`(period_ym, staff_id)` —— 每月每人一条。

**用户决策（2026-04-26）**：
- 历史人力数据后续会陆续补充
- 工号变更视为不同人（不做身份合并）
- 同月重复导入采用"整月覆盖"策略

```sql
CREATE UNIQUE INDEX ux_period_staff ON fact_headcount(period_ym, staff_id);
```

**增量导入工作流**：

```
1. 解析 Excel，识别文件中包含的 period_ym 集合
2. 开启事务 BEGIN
3. 对每个 period_ym：
   DELETE FROM fact_headcount WHERE period_ym = ?
4. 批量 INSERT 新数据（已过滤小计行）
5. 提交事务 COMMIT
6. UI 反馈：
   - 本次导入：N 行（剔除 X 行小计）
   - 覆盖月份：YYYY-MM 列表
   - 当前库覆盖：起始月 ~ 结束月
```

**冲突处理**：
- 同 period_ym 重复导入 → 整月先 DELETE 再 INSERT（事务保证原子）
- 跨月追加 → 不影响已有月份
- 工号在历史月份与新月份不同 → 视为两个不同员工

### 3.3 与保单表的关联校验

导入后建议做一次连贯性检查：

```sql
-- 保单表中存在但人力表中找不到的工号
SELECT DISTINCT staff_id FROM fact_premium
WHERE staff_id NOT IN (SELECT staff_id FROM fact_headcount)
  AND staff_id != '';
```

期望：少量（试用期、临时工等），数量过大说明人力数据缺失月份。

### 3.4 缺失月份提示

启动时检查人力表的 `period_ym` 范围，若与保单表覆盖月份不一致，UI 提示：

```
保单数据覆盖：2024-01 至 2026-04
人力数据覆盖：2026-01 至 2026-04
⚠️ 2024-01 至 2025-12 缺人力数据，活动率/产能模块此期间无法展示
```

## 5. SQLite Schema（v1.1，已确认）

```sql
CREATE TABLE fact_headcount (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  -- 时间
  period_ym  TEXT NOT NULL,        -- '2026-04'
  year       TEXT NOT NULL,
  month      INTEGER NOT NULL,
  -- 组织
  org        TEXT NOT NULL,
  sales_seq  TEXT,
  dept_code  TEXT,
  team_code  TEXT,
  -- 人员
  rank_grade TEXT,
  rank_level TEXT,
  staff_id   TEXT NOT NULL,
  staff_name TEXT,
  tenure_band TEXT,
  biz_mode   TEXT,
  hire_year  INTEGER,
  hire_month INTEGER,
  -- 派生（v1.1 新增）
  mgmt_level TEXT,                  -- '外勤管理职' | '一线营销员' | NULL
  -- 在职
  hc_start   INTEGER NOT NULL DEFAULT 0,  -- 月初 0/1
  hc_end     INTEGER NOT NULL DEFAULT 0   -- 月末 0/1
);

CREATE UNIQUE INDEX ux_hc_period_staff ON fact_headcount(period_ym, staff_id);
CREATE INDEX ix_hc_period_org ON fact_headcount(period_ym, org);
CREATE INDEX ix_hc_staff ON fact_headcount(staff_id);
CREATE INDEX ix_hc_period ON fact_headcount(period_ym);
CREATE INDEX ix_hc_mgmt ON fact_headcount(period_ym, mgmt_level);  -- v1.1 新增
```

**`mgmt_level` 派生规则（待用户在 P3 启动前提供具体职级名单）**：

```js
// 临时实现（用户提供职级名单后替换为字典查表）

  const text = `${rankGrade}${rankLevel}`;
  if (/经理|主管/.test(text)) return '外勤管理职';
  return '一线营销员';
}
```

正式规则需用户提供：哪些 `rank_grade` / `rank_level` 值算「外勤管理职」（见模块 03 Q8）。

## 6. 派生指标定义

基于本表（与 `fact_premium` JOIN）可计算的关键指标。**用户 2026-04-26 决策已落地**。

### 6.1 月末/月初/月均在职人力

```sql
SELECT period_ym, org,
       SUM(hc_start) AS hc_start_total,
       SUM(hc_end)   AS hc_end_total,
       (SUM(hc_start) + SUM(hc_end)) / 2.0 AS hc_avg
FROM fact_headcount
GROUP BY period_ym, org;
```

> 模块 04（活动率）、模块 05（产能）默认采用 **月均**（`hc_avg`）。

### 6.2 活动人力（取代旧称「活跃人力」）

> 用户决策（2026-04-26）：术语统一为「活动人力」，废弃「活跃人力」。

**定义**：当月有保单（zhsf_cents > 0）的不重复员工数。

```sql
SELECT p.year, p.month, h.org,
       COUNT(DISTINCT p.staff_id) AS act_n
FROM fact_premium p
JOIN fact_headcount h
  ON h.staff_id = p.staff_id
 AND h.year = p.year
 AND h.month = p.month
WHERE p.staff_id != ''
  AND p.zhsf_cents > 0
GROUP BY p.year, p.month, h.org;
```

### 6.3 长险活动人力

**定义**：当月有长险保单的员工，是「活动人力」的子集。

```sql
SELECT p.year, p.month, h.org,
       COUNT(DISTINCT p.staff_id) AS chx_act_n
FROM fact_premium p
JOIN fact_headcount h
  ON h.staff_id = p.staff_id
 AND h.year = p.year
 AND h.month = p.month
WHERE p.term_type = '长期险'  -- 实际值待 Q11 确认
  AND p.staff_id != ''
  AND p.zhsf_cents > 0
GROUP BY p.year, p.month, h.org;
```

### 6.4 长险活动率（已确认：分母=月均）

```sql
WITH chx_active AS (
  -- 长险活动人力
  SELECT p.year, p.month, h.org,
         COUNT(DISTINCT p.staff_id) AS chx_act_n
  FROM fact_premium p
  JOIN fact_headcount h
    ON h.staff_id = p.staff_id
   AND h.year = p.year
   AND h.month = p.month
  WHERE p.term_type = '长期险'
    AND p.staff_id != ''
    AND p.zhsf_cents > 0
  GROUP BY p.year, p.month, h.org
),
hc_avg AS (
  SELECT year, month, org,
         (SUM(hc_start) + SUM(hc_end)) / 2.0 AS hc_avg
  FROM fact_headcount
  GROUP BY year, month, org
)
SELECT t.year, t.month, t.org,
       1.0 * COALESCE(a.chx_act_n, 0) / NULLIF(t.hc_avg, 0) AS chx_activity_rate
FROM hc_avg t
LEFT JOIN chx_active a USING(year, month, org);
```

### 6.5 队伍产能（3 个定义，已确认）

| 编号 | 指标 | SQL 概要 |
|------|------|---------|
| **P1** | 人均保费 | `SUM(qj_cents)/100 / 月均在职` |
| **P2** | 人均产能 | `SUM(qj_cents)/100 / 月均活动人力` |
| **P3** | 人均件数 | `COUNT DISTINCT policy_no（不含附加险）/ 月均在职` |

详细 SQL 见模块 05 §5。**注意**：分子统一用 `qj_cents`（期交保费），件数排除附加险（识别规则待 Q10）。

### 6.7 外勤管理职聚合

```sql
SELECT period_ym,
       mgmt_level,    -- '外勤管理职' / '一线营销员'
       SUM(hc_end) AS hc_n,
       (SUM(hc_start) + SUM(hc_end)) / 2.0 AS hc_avg
FROM fact_headcount
GROUP BY period_ym, mgmt_level
ORDER BY period_ym, mgmt_level;
```
